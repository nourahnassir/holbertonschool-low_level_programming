# Green Tech Efficiency & Benchmarking Lab

## Measurement Methodology
To evaluate the algorithmic efficiency and software sustainability of our implementations, we established a controlled instrumentation framework using the standard C `<time.h>` library. The core metrics were captured using the `clock()` function, which measures the CPU time consumed by the program in clock ticks. These ticks were subsequently converted into true elapsed seconds by dividing the difference between the end and start periods by the system-defined `CLOCKS_PER_SEC` constant. For the foundational baseline and instrumentation experiments, static array allocations were utilized to remove memory allocation noise. To ensure reliability and mitigate the effects of background operating system processes, multiple sequential executions were performed on the identical environment, and the averages were extracted. This approach isolates the CPU execution cycles from external variables, providing a trustworthy proxy for overall performance analysis.

## Observed Performance Differences
The empirical data collected across our benchmarking suite revealed staggering performance disparities between the tested paradigms. In the algorithmic comparison experiment, the Naive algorithm (`count_even_naive`) evaluated a dataset of 50,000 integers using a nested-loop architecture ($O(N^2)$), resulting in an average execution time of ~0.652156 seconds. Conversely, the Single-pass algorithm (`count_even_single_pass`) processed the exact same data using a single linear loop ($O(N)$), yielding an average execution time of ~0.000104 seconds. This represents an astronomical relative difference, where the Naive implementation is approximately 6,270 times slower than the optimized Single-pass equivalent. Furthermore, in the multi-phase instrumentation laboratory, the `PROCESS` phase—which involves extensive arithmetic logic—consumed ~0.000342 seconds, which was the largest sub-component of the `TOTAL` execution runtime (~0.000789 seconds) compared to `BUILD_DATA` (~0.000282 seconds) and `REDUCE` (~0.000165 seconds).

## Relation Between Runtime and Energy Consumption
In green computing and software engineering, processor runtime acts as a direct indicator of carbon footprint and total power draw. Hardware components like the central processing unit (CPU) operate at specific wattage rates when active. When an inefficient algorithm like our Naive implementation forces the CPU to run for ~0.652156 seconds instead of ~0.000104 seconds, it holds the processor in a high-power state for a substantially longer duration. Longer processing durations lead directly to higher total energy consumption measured in Joules (Watt $\times$ Seconds). On a massive scale, such as cloud data centers housing millions of operations, switching from $O(N^2)$ to $O(N)$ algorithms cuts down continuous cooling requirements, electricity costs, and greenhouse gas emissions.

## Limitations of the Experiment
While this laboratory provides vital comparative observations, it exhibits inherent structural limitations. First, using the `clock()` function is highly dependent on system scheduling and thread interruptions, which introduces minor variability between sequential runs. Second, runtime is used only as a proxy; the experiment lacks direct hardware power profiling tools (such as Intel RAPL or hardware voltmeters) to measure real-time power draw in Watts. Finally, the static dataset size of 50,000 elements is relatively tiny and fails to simulate memory cache misses, page faults, or hardware throttling constraints that occur under severe, long-term enterprise production workloads.

## Practical Engineering Takeaway
The ultimate takeaway from this benchmarking lab is that architectural and algorithmic choices represent the primary defense against computational waste. Engineers must prioritize selecting the correct Big-O complexity classes during early design phases rather than attempting micro-optimizations later. Writing computationally lean code is no longer just an abstract goal for speed; it is an essential engineering responsibility for environmental and hardware sustainability.
